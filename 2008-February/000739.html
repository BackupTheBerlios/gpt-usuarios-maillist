Se possível eu tentaria evitar que código em Portugol (e consequentemente a
VM) tivesse que lidar com as estruturas definidas em C. Em último caso que
sejam criadas funções C que manipulam esses dados e a VM invoca essas
funções. Eu acho que colocando manipulação de estruturas de C dentro da VM e
do portugol vai sobrecarregar o projeto agora. Imagina que a gente vai ter
que ter suporte a todos os tipos de C dentro do Portugol e da VM. Não quero
dizer que não seja útil, mas eu tentaria evitar isso nesse momento.

Minha sugestão: se realmente for a melhor forma, então que a gente deixa
para a próxima versão, e colocamos na Gepeto o que for necessário para
mascarar essa dependência.

Ok. Mas acho que alguns mapeamentos são certos, então podem estar fixos em
algum ponto. No exemplo acima, const char * = literal sempre será válido
(pelo menos eu acho isso).

Outra questão é o triplo propósito da assinatura de uma rotina no sld:
>
>
> function fopen(string name, string mode) : pointer[FILE]


Faltou a definição da função para mapeamento em C:

function fopen(string name, string mode) : pointer[FILE] ::= fopen(name,
mode);

Esta declaração deve informar, principalmente:
>
> 1) a assinatura da função/procedimento portugol para ser registrada na
> tabela de simbolos;


ok

2) a variável/tipo C/C++ que representará os valores na pilha
> (parâmetros e resultado);


ok

3)  o método da pilha que deve ser utilizado para recuperar/armazenar
> determinado parâmetro/resultado.


Fiquei com dúvida neste ponto. Isso realmente precisa ? Conforme o tipo de
dado, tem-se o método de push/pop equivalente.

Agora, supondo que os termos entre [] denotam o conjunto de métodos
> para as diferentes operações pop/push da pilha (popString, popInt,
> ...), eu pensei no seguinte esquema:
>
>
> function fputs(cont char* text [popString], FILE* file [popint]) : int
> [pushInt];


Nessa abordagem, ao invés de colocar os tipos do Portugol, foram colocados
os tipos da linguagem nativa do bind (nesse caso C). Ñão sei se foi
intencional a troca, mas acho que essa troca foi válida sim.

Porém o que me preocupa é a sintaxe de C. Se a gente deixar a definição do
sld nos mapeamentos com tipos de C, a gente vai ter que lidar com static,
const, pointer, ... enquanto que se os tipos forem os de Portugol, o parser
é bem simples.

function fputs(literal text, ponteiro file) : inteiro ::= fputs(text, file);


Com esta declaração e as informações de tipos nas seções acima,
> poderiamos derivar a assinatura gpt (problema 1) e demais definições
> de tipos, completando as informações necessárias no lado do portugol:


Até podemos derivar a assinatura, mas mesmo que seja opcional na sintaxe do
sld, eu acharia melhor que fosse possível indicar a ordem dos parâmetros na
chamada a uma função C. Se a função C recebe parâmetro A e B, mas a gente
achar mais conveniente em portugol o usuário invocar com B e depois A, fica
fácil a gente trocar. Isso também é válido caso a gente queira trocar o nome
de uma ou mais callbacks...

Mesmo que seja opcional na gramática do sld, seria interessante podermos
indicar a função em C ou em Portugol e indicar a ordem das chamadas.

Estes exemplos acima lidam apenas com associação de tipos, mas ainda
> não lidam com construção. Supondo que a biblioteca C tenha  estruturas
> como:
>
> struct Point {
> int x;
> int y;
> };
>
> struct Rectangle {
> Point topleft;
> Point bottomright;
> }
>
> Para mapear estes tipos precisariamos de uma notação para construção
> de estrutura. Se a sintaxe do portugol for inconveniente, talvez algo
> como isso:
>
> gpt_types {
>
> //mapeamentos
>
> FILE=ponteiro;
>
> //construções
>
> struct(Point) = x:inteiro, y:inteiro;
> struct(Rectangle) = topleft:Point, bottomright:Point;
> }


Eu tenho muito receio de tentarmos criar structs em Portugol equivalentes as
de C, até porque a gente não tem hoje os mesmo tipos de C. Imagine que um
dia a gente queira fazer bind com outra linguagem. Seguindo esse estilo, se
não estou enganado, teremos que ter suporte em Portugol aos tipos de dados
da outra linguagem também. Eu prefiro criar equivalências com Portugol do
que ter suporte aos tipos de dados de C. O que não tem equivalência a gente
mascara.

A minha proposta fica em tentar simplificar e, se necessário uma estrutura,
fazer algo como indicar o tamanho da estrutura. O que "tem dentro" fica
abstraído em Portugol. A VM apenas precisaria saber o tamanho da estrutura
para pode copiar os dados internos. E quanto ao indicar o tamanho, pode até
ser algum código que quando a VM inicializa ela pega o sizeof da estrutura e
mantém para saber lidar com a informação.

Mas vamos conversando para podermos sincronizar as idéias....
