Eu gostei da questão de usar opcodes (assembly) da VM na abordagem 1. Porém,
pelas idéias iniciais, a gente não tem um opcode que reflete cada
funcionalidade do S.O. porque senão teríamos uma combinação gigantesca de
opcodes, além de termos uma VM específica para um conjunto de bibliotecas do
sistema.

Eu acharia interessante utilizar a abordagem 1, mas ao invés de termos um
opcode para cada funcionalidade de uma biblioteca, poderíamos ter um opcode
genérico que consiga invocar as funções previamente mapeadas das bibliotecas
do sistema. Algo como:

push "cria" // função
push "thread" // "classe"
libcall

De quaquer forma, quanto a essa questão de termos ou não opcodes específicos
para threads, IO e afins, acho que temos que conversar melhor

Para não termos uma VM específica para uma ou mais bibliotecas, a gente
poderia ter um opcode que cria uma janela, por exemplo, independente da
biblioteca gráfica que está sendo utilizada. Porém, de qualquer forma,
teríamos uma combinação bem grande de opcodes, o que eu acho que não é uma
boa abordagem. Apesar disso, acho que a gente deveria investigar melhor essa
sugestão.
