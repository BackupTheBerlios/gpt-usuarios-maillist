É uma abordagem utilizar dlopen. Outra possível abordagem seria o call, ao
retirar os parâmetros de biblioteca e função da pilha (os 2 ld's) invocaria
a função correta. Um rascunho de código seria algo como:

map<string,(*void)()> call_opcodes;
// pode ser hash_map ou alguma variante... não se preocupem muito com o
código no momento porque é só um exemplo...

call_opcodes["LIBC.FOPEN"] = bind_LIBC_FOPEN;
call_opcodes["B2_F2"] = bind_B2_F2;

void opcode_call( )
{
    string biblioteca = pilha.pop( );
    string funcao = pilha.pop( );

    (*void)() bind = call_opcodes[biblioteca+"."+funcao];
    (*bind)(); // se eu errei a sintaxe, desculpem-me, mas aqui eu estou
invocando a funcao que está no ponteiro;
}

void bind_LIBC_FOPEN( )
{
    // Retira os seus parametros da pilha...
    FILE* file = fopen( xxx, xxx );
    // Insere o valor de retorno na pilha (file)...
}

void bind_B2_F2( )
{
    ...
}

E por ai vai. Podemos fazer um rascunho de código com dlopen também.

Agora, imagina gerar o map acima e um método para cada função de cada

> unica diferença, então, entre as duas abordagens seria apenas o número
> de opcodes para realizar a operação (1 x 2), é isso?


Se só existir o opcode call, então teríamos algo como os seguintes
opcodes internamente na VM:

const int OPCODE_NOP = 0;
const int OPCODE_LD = 1;
const int OPCODE_ST = 2;
const int OPCODE_CALL = 3;
...

Se, para cada função criarmos um opcode, então teríamos algo como:
 const int OPCODE_NOP = 0;
const int OPCODE_LD = 1;
const int OPCODE_ST = 2;
const int OPCODE_CALL_LIBC_FOPEN = 3;
 const int OPCODE_CALL_B2_F2 = 4;
const int OPCODE_CALL_B2_F3 = 5;
const int OPCODE_CALL_B2_F4 = 6;
...

Ou seja, para cada nova função seria gerado um novo opcode. Com isso, para
incluir uma função na linguagem, teríamos que alterar também a geração
de´bytecode do compilador.

De fato, acho melhor ter opcodes enxutos. Com um pequeno conjunto,
> fica fácil aprender e brincar na VM, em todos os aspectos. Para
> performance, acho razoável deixar a cargo dos otimizadores, jitter,
> inlining, etc. E de qualquer forma, não acho que teriamos necessidade
> de criar opcodes para otimizar certas operações.


Até podemos criar um simples, do tipo:

 Ao invés de LD 1, ter um opcode LD_1...
Ao invés de LD 2, ter um opcode LD_2...
Ao invés de LD 3, ter um opcode LD_3...

A JVM tem esses opcodes para os números de 0 a 5, se eu não me engano. Mas
também é detalhe e se não tivermos ninguém vai morrer hehehe. A única
utilidade prática disse é para aplicar otimizações de código, ou você ver
que, se gerasse o assembly "na mão", faria melhor do que o seu compilador
rsrsrs.
