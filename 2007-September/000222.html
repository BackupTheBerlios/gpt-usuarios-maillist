Oi pessoal.

Fiz uma pesquisa rápida (4 horas) sobre algumas VM's existentes. A intenção
desta pesquisa era averiguar o que está sendo desenvolvido atualmente.

Quanto a tipagem, existem 2 tipos de VM's:
* Fortemente tipadas, geralmente utilizadas para gerar código a partir
de linguagens tipadas como pascal, java, C;
* Fracamente tipadas, geralmente utilizadas por linguagens fracamente
tipadas ou que possuem tipagem dinâmica como as linguagens scripts.

Em geral, todas as VM's tem algum suporte a pilha de dados para poderem ter
funções recursivas, passagem de parâmetros, entre outras coisas.

Existe uma divisão clara entre as VM's stack-based (baseada em pilha) e as
register-based (baseada em registradores). Atualmente parece haver uma
tendência no desenvolvimento de VM's baseada em registradores. Os motivos
basicamente são porque geram um bytecode menor e são mais eficientes. Porém,
o número de opcodes dessas VM tende a ser maior.

Outro ponto em que eu subdividiria as VM's é entre as que tem consciência
dos tipos de dados que manipulam e as que não tem consciência disso. As que
sabem os tipos de dados que manipulam são a maioria dentre as que olhei.
Geralmente, as que tem consciência dos dados, tendem a ter um conjunto de
operações (opcodes) maior.

Quanto a utilizar código intermediário, pelo que notei, são poucas que
utilizam. Em geral ou se gera o arquivo contendo o bytecode (processo mais
trabalhoso) ou se gera um arquivo texto contendo assembly da VM e um
montador (assemblador) transforma o assembly em bytecode.

As VM's que eu vi utilizando código intermediário são projetos que tem, como
meta, serem uma VM "universal" (talvez tenha exagerado na palavra). Nesses
projetos a idéia é que outros compiladores, de variadas linguagens, poderiam
gerar um CI e, a partir deste CI, as ferramentas disponibilizadas junto com
a VM traduziriam esse CI para assembly e bytecode.

Quanto a nossa VM, pelo que temos hoje, podemos dizer que:
* Ela é parcialmente baseada em registradores (vamos ampliar isso, eu acho);
* Tem pilha apenas para gerenciamento de funções e procedures;
* Não tem consciência "absoluta" dos tipos de dados que está manipulando; a
VM sabe o que é um int, mas se for gerado um código que tente atribuir um
int para um tipo de dado string, ela vai executar;
* Temo um CI definido.

Vou colocar abaixo alguns itens para discussão:

1) Provavelmente o CI será retirado do projeto (minha opinião) porque não
temos como meta, não que eu saiba, sermos uma VM "universal". A gente quer
dar suporte ao gpt apenas;
2) Quanto ao uso de registradores, provavelmente vamos ampliar isso para que
o bytecode fique mais "limpo";
3) Quanto a nossa VM ter ou não consciência dos dados que está manipulando,
eu diria que inicialmente ela foi desenvolvida para ser uma VM sem
consciência dos dados, mas atualmente não sei dizer se essa seria a melhor
abordagem. Talvez seja interessante colocarmos um pouco mais de inteligência
na VM.

Por enquanto era isso :-)
