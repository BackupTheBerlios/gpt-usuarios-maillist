O que acontece: O Antlr gera um mapa (dict) de palavras-chave que foi
definido no lexer.g. É um mapa de string (palavra chave) e int (token
correspondente a palavra chave). O nome deste mapa é "literals". Ele é
utilizado para fazer comparações com o texto do algoritmo do usuário
para encontrar o token correspondente das palavras chave. No
PortugolLexer ele é inicializado assim:

literals["algoritmo"] = 3;
literals["início"]    = 4;
....etc


Uma vez que estas palavras chave utilizam acento e o lexer.g (onde
elas são definidas) está em iso-8859-1, se o usuário passar um
algoritmo em utf-8, a comparação de strings com acento vai falhar,
pois a codificação de "início" no algoritmo e "início" do lexer é
diferente.

Daí, eu tinha passado a madrugada tentando manter os dois encodings.
Consegui achar funções para descobrir o encoding do sistema
[nl_langinfo(CODESET)], testei aqui no slack e no kubuntu, e deu
certo. Depois, penei tentando utilizar as funções da libconv para
fazer conversão de textos.

Como o PortugolLexer chama "initLiterals" no construtor, que cria o
mapa "literals", eu tinha criado um "initLiterals_utf8", que cria mais
entradas no dicionário, de palavras convertidas. Mais ou menos isso,
em pseudocodigo:

foreach keyword, token in literals
  literals[to_utf8(keyword)] = token;

O que criaria, na tabela, duas entradas para cada palavra chave: uma
em iso-8859-1 (que é o encoding do código fonte), e outra em utf8,
ambos com o mesmo valor de token.

Mas isso não foi suficiente. Além de ter que produzir uma tabela de
literais na codificação que o usuário está utilizando, o processo do
scanner tem que ser diferente para utf-8. Foi ai que utilizei o
UnicodeCharScanner, dos exemplos do antlr.

Eu utilizei este scanner tendo o PortugolLexer como subclasse dele. O
problema é que se o usuário passar um algoritmo em iso-8859-1, este
scanner não vai funcionar. Ele teria que passar o trabalho para o
CharScanner, e utilizar o InputBuffer ao invés de UnicodeCharBuffer.

Então, eu tenho que resolver qual vai ser a superclasse de
PortugolLexer em tempo de execução (CharScanner ou
UnicodeCharScanner). Até que dá pra fazer isso com c++ templates (e
talvez, isso desdobraria em outros problemas), mas eu defino
PortugolLexer na gramática lexer.g, e não dá pra fazer isso lá.

Além de tudo isso, o GPTDisplay também teria que ser modificado (mas
algo bem mais simples) para exibir erros e informações no encoding do
sistema.

Depois dessa novela toda, joguei fora todas essas modificações e
simplesmente passei o iconv em todos os códigos fontes do GPT,
convertendo eles para utf-8. Isso, somado a utilização do
UnicodeCharScanner como superclasse do PortugolLexer permitiu que
algoritmos utf-8 fossem processados. Mais detalhes sobre as
modificações comitadas, é só ver o diff da revisão anterior com a
revisão atual.
