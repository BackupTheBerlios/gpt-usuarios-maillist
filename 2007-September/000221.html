Não vejo problema nisso. A questão que levantei quanto a gerar um pcall
imprima para cada parêmetro é que, no CI (ou CTE), tudo tem que ser
extremamente simples, nada de construções de linguagens complexas. O
agravante, claro, é que o imprima, além de ter uma lista de argumentos
variáveis, podem ser de tipos diferentes. Mas enfim, no caso do CTE, um
pcall para cada argumento resolve o problema (minha opinião, claro).

Se sim, devemos definir a sintaxe e semântica
> para funções que utilizam parâmetros variáveis. Caso contrário,
> possibilitar overriding de funções baseado nos tipos de argumentos
> seria suficiente, na minha opnião:
>
> função imprima(i : inteiro) ...
> função imprima(s : literal) ...
> função imprima(r : real) ...


Isso mesmo. A idéia inicial era que, durante a geração do CTE, não seria
necessário se preocupar com a função imprima a ser invocada
(imprima_inteiro, imprima_caracterer, ...). Isso seria preocupação durante a
geração do assembly.


> imprima("2 + 2 = ");
> imprima(2 + 2);
> imprima("\n");
> //Imprime: 2 + 2 = 4


Exato. Só que a gente pode deixar o usuário passar um número de parâmetros
variáveis no imprima. O problema é resolvido invocando a função imprima para
cada um dos argumentos.

Para resolver a legibilidade, poderiamos utilizar um operador de
> concatenação capaz de operar sobre qualquer tipo de dados. Então:
>
> imprima("2 + 2 = " >> (2 + 2) >> "\n"); //O operador >> é apenas
> ilustrativo


Ai eu já não sei dizer o que é melhor.
