Boiei agora rsrsrs.

Talvez eu não tenha explicado corretamente ou eu realmente não entendi o que
você disse. Vou tentar explicar melhor então.

A minha idéia é que a gente pode desenvolver um gerador de código para
bindings de bibliotecas C (simples, a princípio) que ele receba como entrada
um mapeamento gpt <-> lib e o template para geração de código, se
necessário. Baseado nisso ele gera o código C/C++ necessário para validar os
tipos, converter, e invocar a biblioteca de sistema. Esse código gerado a
gente acrescentaria na VM.

Vou mostrar um exemplo assumindo que a VM terá apenas um opcode para invocar
qualquer função de qualquer qualquer biblioteca. Mesmo que tenhamos um
opcode para cada função de cada biblioteca, nós teremos um problema similar.

algoritmo teste;
use B1,B2; // chutei uma sintaxe
início
    F1( ); // Essa função está na biblioteca B1
    F2( ); // Essa função está na biblioteca B2
fim

Se errei a sintaxe do gpt, desculpem-me :-)

Esse trecho de código geraria o seguinte trecho em assembly:

ld "B1"
ld "F1"
call

ld "B2"
ld "F2"
call

Neste exemplo existiria apenas um opcode (call) para invocar qualquer função
de qualquer biblioteca.

A princípio, nós teríamos que escrever código C++ (no braço) para validar os
tipos (conforme a função sendo invocada), chamar a função correta da
biblioteca, entre outros (pode ser uma tabela com mapeamentos). Agora
imagine a quantidade de funções que a gente terá que implementar. Além
disso, em alguns casos, uma função do gpt pode invocar várias funções de uma
lib. Fazer tudo isso na "munheca" vai dar muito trabalho.

A gente poderia ter um arquivo que contivesse os mapeamentos entre gpt e as
libs. O gerador de "bindings" para bibliotecas lê esse arquivo e gera o
código C++ que será acrescentado ao código da VM.

Exemplo de arquivo para os bindings:

inteiro abre_arquivo (string nome) ::= int open(string nome, "r+")

inteiro fecha_arquivo(inteiro arquivo) ::= int close(arquivo)
