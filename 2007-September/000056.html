On 9/6/07, Alex Garzao <alexgarzaol em gmail.com> wrote:
> Já que eu mencionei sobre um ou N opcodes para invocar uma função no outro
> e-mail, vou esboçar aqui duas alternativas:
>
>
> algoritmo teste;
> use B1,B2; // chutei uma sintaxe
> início
>     F1( ); // Essa função está na biblioteca B1
>     F2( ); // Essa função está na biblioteca B2
> fim
>
> Utilizando sempre um opcode para invocar qualquer função de qualquer
> biblioteca, teríamos o seguinte trecho em assembly:

Só para constar, o que chamas de "biblioteca" aqui, é uma lib externa?
Então, B1, por exemplo, poderia ser pthread? E F1, poderia ser
pthread_create? É isso?

> ld "B1"
> ld "F1"
> call


Perdoe se meu raciocínio for falho:
"ld" expandiria para algo, efetivamente, como "dlopen", correto? A
unica diferença, então, entre as duas abordagens seria apenas o número
de opcodes para realizar a operação (1 x 2), é isso?

De fato, acho melhor ter opcodes enxutos. Com um pequeno conjunto,
fica fácil aprender e brincar na VM, em todos os aspectos. Para
performance, acho razoável deixar a cargo dos otimizadores, jitter,
inlining, etc. E de qualquer forma, não acho que teriamos necessidade
de criar opcodes para otimizar certas operações.
