On Dec 18, 2007 7:38 AM, Alex Garzao <alexgarzaol em gmail.com> wrote:
> Posso estar enganado, mas acho que uma abordagem era transformar tudo para
> lógico (ou inteiro, como preferir). O GptC geraria, a partir de cada tipo de
> dado não lógico, o seu equivalente lógico.
>
> Dessa forma acho que a primeira opção seria a mais flexível.
>
> Algo como:
>
>
> x : real;
> y : string;
> z : lógico
>
> z := x ou y; //ok
> z := não y; //ok
>
> // GptAssembly
> var x real
> var y string
> var z bool
> var t1 bool
> var t2 bool
> var t3 bool
>
> rne t1, x, 0 // verifica se x eh zero e armazena em t1 o resultado
> sne t2, y, ""
> ine t3, z, false
>
> or z, t1, t2
> not z, t2
>
> Acho que seria algo similar a isso...
>

É,  eu pensei nisso tbm, pois seria complicado ter um opcode para cada
combinação de operandos.

Agora, queria a opnião de vcs para saber se vamos deixar esses
operadores mais flexíveis quanto aos tipos, ou se vamos exigir um
controle maior sobre os tipos operandos.
