>
> Ok, vamos rever e discutir um pouco as operaçõs lógicas.
>
> Bitwise não há mistério (&, |, ^, ~). Os operandos são do tipo inteiro
> e o resultado é sempre do tipo inteiro.
>
> Mas e as operações lógicas (e, ou, não), como devem funcionar?
>
> Eu imagino que podemos escolher 3 opções:
>
> 1-Os operandos podem ser de qualquer tipo, e podem ser de tipos
> diferentes um do outro:
>
> x : real;
> y : string;
> z : lógico
>
> z := x ou y; //ok
> z := não y; //ok
>
> 2-Os operandos devem ser do mesmo tipo:
>
> x : real;
> y : real;
> k : string;
> z : lógico;
>
> z := x ou y; //ok
> z := não k; //ok
> z := x ou k; //erro
>
>
> 3-Os operandos só podem ser do tipo "lógico":
>
> x : lógico;
> y : lógico
> z : lógico;
>
> z := x ou y; //ok
> z := não x; //ok
>
> Vc tem alguma opnião sobre como deve ser?
>

Posso estar enganado, mas acho que uma abordagem era transformar tudo para
lógico (ou inteiro, como preferir). O GptC geraria, a partir de cada tipo de
dado não lógico, o seu equivalente lógico.

Dessa forma acho que a primeira opção seria a mais flexível.

Algo como:

x : real;
y : string;
z : lógico

z := x ou y; //ok
z := não y; //ok

// GptAssembly var x real
var y string
var z bool
var t1 bool
var t2 bool
var t3 bool

rne t1, x, 0 // verifica se x eh zero e armazena em t1 o resultado
sne t2, y, ""
ine t3, z, false

or z, t1, t2
not z, t2

Acho que seria algo similar a isso...

De qq forma achei estranho um "ou" com string e variável, mas acho que para
os usuários isso será intuitivo.
