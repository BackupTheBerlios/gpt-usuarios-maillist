Ok, agora que tive tempo de dar uma olhada nos bindings. Alguns pensamentos...


Da forma acima, isso resolveria o problema para ponteiros, mas creio
que precisamos de algo mais completo para o mapeamento de tipos.


Abrindo parenteses, em se tratando do mapeamento da estrutura FILE, eu
pensei em expandir seu tipo para uma estrutura portugol, mas pelo que
vi do padrão, os campos dela não são padronizados, então, acho que
deixariamos sua estrutura interna inacessível a nível de portugol
mesmo. Por outro lado, para uma estrutura como Elf32_Ehdr (em elf.h)
por exemplo, deveriamos fazer o mapeamento para um tipo "estrutura".

Prosseguindo, a meu ver, o arquivo sld poderia ter uma seção para
lidar com a associação de tipos (possivelmente, poderia ser uma seção
similar à "options{}"). Mas, pensando bem, em relação aos tipos, acho
que lidamos com duas atividades distintas: criação de tipos portugol e
associação de tipos gpt<->C. Então, proporia termos locais específicos
para lidar com estas duas tarefas.

Um rascunho inicial:


//file.sdl
gpt_types {
  //tipo FILE é ponteiro;
  FILE=ponteiro;
}

type_mappings {

  //C_type=GPT_type

  const char*=literal;
  char*=ref literal;
  FILE*=FILE; //tipo gpt definido acima
  int=inteiro;
}


"gpt_types" ajudaria a criar a tabela de símbolos da biblioteca.
"type_mappings" ajuda no mapeamento de tipos gpt<->c.


Outra questão é o triplo propósito da assinatura de uma rotina no sld:


function fopen(string name, string mode) : pointer[FILE]


Esta declaração deve informar, principalmente:

1) a assinatura da função/procedimento portugol para ser registrada na
tabela de simbolos;
2) a variável/tipo C/C++ que representará os valores na pilha
(parâmetros e resultado);
3)  o método da pilha que deve ser utilizado para recuperar/armazenar
determinado parâmetro/resultado.

Agora, supondo que os termos entre [] denotam o conjunto de métodos
para as diferentes operações pop/push da pilha (popString, popInt,
...), eu pensei no seguinte esquema:


function fputs(cont char* text [popString], FILE* file [popint]) : int
[pushInt];



Com esta declaração e as informações de tipos nas seções acima,
poderiamos derivar a assinatura gpt (problema 1) e demais definições
de tipos, completando as informações necessárias no lado do portugol:


tipo FILE é ponteiro;
função fputs(literal, FILE) : inteiro


E podemos gerar as variáveis e tipos C/C++ que representam o
determinado parâmetro (problema 2), informar o método de push/pop
(problema 3) e, por fim, gerar a operação:

const char * ctext = dataStack.popString().c_str();
FILE* cfile = (FILE*) dataStack.popInt();
int result;
result = fputs(ctext,cfile);
dataStack.pushInt(result);


Fatorando este esquema, poderiamos colocar as informações do método
push/pop a serem utilizadas na seção type_mappings. Assim:

type_mappings {
  [string] const char*=literal;
  [string] char*=ref literal;
  [int] FILE*=FILE;
  [int] int=inteiro;
}

Onde o que está entre [] indica o método push/pop. Agora, a nova
definição de fputs:

function fputs(cont char* text, FILE* file) : int;



Estes exemplos acima lidam apenas com associação de tipos, mas ainda
não lidam com construção. Supondo que a biblioteca C tenha  estruturas
como:

struct Point {
 int x;
 int y;
};

struct Rectangle {
  Point topleft;
  Point bottomright;
}

Para mapear estes tipos precisariamos de uma notação para construção
de estrutura. Se a sintaxe do portugol for inconveniente, talvez algo
como isso:

gpt_types {

  //mapeamentos

  FILE=ponteiro;

  //construções

  struct(Point) = x:inteiro, y:inteiro;
  struct(Rectangle) = topleft:Point, bottomright:Point;
}
