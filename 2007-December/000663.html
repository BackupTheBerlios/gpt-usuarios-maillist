Oi Thiago,

Eu utilizei praticamente todo aquele código que você tinha para uso de
syslibs. Adaptei alguma coisa na VM apenas.

Para facilitar a nossa vida, a gente teria que desenvolver as libs dinâmicas
(.so) em C++ porque aproveitamos todas as nossas classes. Mas aí já são
outros 500...

Um problema é que a gente não consegue invocar um construtor ou destrutor
que esteja definido dentro de um .so. Para isso se resolve criando duas
funções C com extern "C" que internamente elas invocam o construtor e o
destrutor.

Os métodos da classe dentro do .so, após isso, podem ser invocados
normalmente.

Bom, até onde eu sei, criar .so em C++ é isso. Se você souber uma forma
melhor eu estou muito interessado em saber como :-)

Essa abordagem é correta quando a gente carrega .so e sabe previamente quais
os métodos que serão invocados. Porém, para nós, isso não elimina todos os
problemas porque nós não sabemos previamente quais os métodos que serão
invocados. O dlsym não consegue encontrar os métodos C++ (por isso das
funções C para o construtor e destrutor). Uma forma seria, para cada método,
criar uma função C equivalente.

Apesar de uma ferramenta nossa ser responsável por gerar esse código, eu
achei essa abordagem danadinha hehehe.

Você conhece alguma forma descente de fazer isso ??? Tem como o dlsym
encontrar os métodos de uma classe ?

O rascunho de código para criar o .so que eu tenho até o momento é este:

include <stdio.h>

#include "CBaseSyslib.hpp"

class teste : CBaseSyslib
{
   void imprime_hello(int i, int j)
   {
      puts ("Hello !!!");
   }
};

extern "C" {
   CBaseSyslib* create_teste()
   {
      return new teste();
   }
   void destroy_teste(CBaseSyslib* obj)
   {
      delete obj;
   }
   void imprime_hello(CBaseSyslib* obj, int i, int j)
   {
      obj->imprime_hello(i, j);
   }
}
