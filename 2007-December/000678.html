> -Qual o motivo do flag no tipo string?


Inicialmente não exisita. A proposta era ter um ponteiro para um tipo string
(quando variável) ou ter a constante em si. Porém, dessa forma, não teria
como saber se os 4 primeiros bytes do endereço da string são uma constante
ou variável. Por isso foi inserido a flag. Quanto constantes temos a flag e
mais a constante terminada em NULL. Quando variável temos a flag e um
ponteiro para um tipo std::string (não estou prevendo UTF8 por enquanto).

Uma opção era diferenciar strings de constantes nos opcodes, mas achei que
isso daria mais trabalho do que colocar a flag.

Pensei também em sempre utilizar um pointer, ou seja, mesmo os dados
constantes seriam armazenados por um ponteiro. Mas daí a VM quando
inicializasse teria que saber quem seriam as contantes para criar o
ponteiro, etc e tal. Não gostei da idéia...

Se eu fiz uma boa escolha, bom, isso já é outra história hehehe.

-Quando um "data" é empilhado, só o endereço é empilhado certo
> (verifiquei em pushdrOpcode)?


Não. Os dados em si são empilhados. Vê se tu está com a última versão da VM.


Aliás, esse opcode nem existe mais :-)

Eu refiz algumas coisas de forma que não é mais necessário empilhar o espaço
para o resultado antes de invocar a subrotina.

-E quando uma matriz é empilhada?


Matriz sempre é ponteiro. Quando empilhada é criada uma cópia da matriz e o
ponteiro da cópia é empilhado. Sim, temos um memory leak aqui :-)

Eu só consigo imaginar três alternativas para implementar o OR / AND lógico:
>
> -Ou todos os dados na pilha tem 32 bits, e só precisamos dos dois
> opcodes: AND e OR lógico.
> -Ou a pilha armazena dados de tamanhos arbitrários, e precisamos de
> várias versões dos opcodes AND e OR lógicos, de acordo com os tipos
> dos operandos.
> -Ou todos os dados são "tagueados", com um cabeçalho indicando seu
> tamanho na pilha.
>
> Vc tem alguma idéia?


Você está falando de AND / OR bitwise ou lógico ?

Para bitwise eu só tinha previsto AND / OR para o tamanho de um inteiro. Se
existir a necessidade de AND / OR bitwise para outros tipos de dados então
teremos que ter uma variante de AND / OR para cada tipo de dado.

Se você está falando de AND / OR lógico, não esqueça que os opcodes de
comparação, maior que, menos e afins, testam dois valores e retornam um
inteiro com o resultado da comparação. Por isso só temos AND / OR lógico
sobre inteiros.

Exemplo em GptAssembly (sem muito nexo, só para mostrar os opcodes):

var s1 string
var s2 string
var t1 int
var t2 int
var t3 int

...
seq t1, s1, s2 // t1 recebe o resultado da comparação
sle t2, s1, s2
or t3, t1, t2
...
