> Tendo em vista que gptassembly não retém todas as informações de tipos
> alto nível, de que forma a tabela de símbolos será passada do
> compilador para o gptasm até o .gvm?


Eu acharia mais interessante a gente seguir uma abordagem tradicional onde o
compilador gera código para o montador, e o montador gera o binário, e ambos
são independentes. Com isso o usuário pode programar em portugol ou
GptAssembly diretamente.

Todas as informações que o montador deve ter tem que estar no seu assembly,
sem nenhuma interferência posterior do compilador. Sim, podem estar faltando
coisas, mas dai a gente conversa e corrige.

Dessa forma o montador gera novamente a sua tabela de símbolos baseado na
declaração de dados (parâmetros, variáveis, procedimentos, ...)

Eu me refiro à "constness" dos identificadores, nomes dos campos em
> uma estrutura, assinatura de rotinas, etc. O gptc será linkado ao
> gptasm, e após este último gerar o gvm, o gptc injetará a tabela de
> simbolos?


O que seria a "constness" dos identificadores ? Dados constantes ???

Estruturas não existem no GptASM, e pouco menos na VM. Uma estrutura, em
GptASM, é representada por uma sequência de bytes (tipo data). Para acessar
cada "membro" devem ser usados os mnemônicos como igetv, sgetv e afins que
realizam acesso indexado.

Algo como:

// Portugol
algoritmo teste;

estrutura coordenada
    x: inteiro;
    y: inteiro;
fim-estrutura

variável c: coordenada;

início
    c.x := 10;
    c.y := 20;
    imprima( "c.x=", c.x );
    imprima( "c.y=", c.y );
fim


// GptAssembly

program exemplo

var c data[8] // sizeof c

proc main
    var t1 int

    // c.x := 10;
    isetv c:0, 10 // *( c + 0 ) := 10

    // c.y := 20
    isetv c:4, 20 // *( c + 4 ) := 20

    // imprima( "c.x=", c.x );
    igetv t1, c:0 // t1 := *( c + 0 )
    pushiv t1
    pushit
    pushsv "c.x="
    pushst
    push_2
    lcall imprima

    // imprima( "c.y=", c.y );
    igetv t1, c:4 // t1 := *( c + 4 )
    pushiv t1
    pushit
    pushsv "c.y="
    pushst
    push_2
    lcall imprima

    exit_0
endproc

endprogram


Quanto a assinatura de métodos, como o GptASM praticamente não tem análise
semântica, sempre que é invocado um procedimento assume-se que o usuário
está chamando corretamente. Eu acredito que através do use (também tem o use
em GptAssembly) e da tabela de símbolos o GptASM conseguirá fazer todas as
consistências necessárias. Se bem que, em GptAssembly, para saber se o
usuário passou os dados corretamente na invocação de uma subrotina não é tão
fácil assim.

Em uma assembly qualquer, eu acho que o montador não tem condições de fazer
esse tipo de consistências. Ou tem ???
