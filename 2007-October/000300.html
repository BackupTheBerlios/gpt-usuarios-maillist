Primeiro um background:

O antlr é uma ferramenta que gera codigo para compiladores. Dado uma
gramatica escrita em uma linguagem própria (similar ao EBNF, mas
extendida), o antlr gera o código para o lexer, parser e treewalkers.
Um dos diferenciais é que ele gera código para diferentes linguagens
(targets). Isso é, dada uma gramática T:

//--------
grammar T;
programa : "ola" "mundo";
//--------

O antlr pode gerar o código correspondente para diferentes linguagens,
que representam esta gramática. Isso é relevante na escolha da
linguagem porque, em geral, as gramáticas misturam código da linguagem
do ANTLR e código da linguagem alvo:

//--------
grammar T;
options {language = Python; }

programa : "ola" "mundo"; { print "finalizado" }
//--------

O código entre chaves (print ) é código python. Então, o código da
gramática acaba ficando muito ligado à linguagem alvo utilizada.

Na versão 2, pode-se gerar código para c++, c#, Java e python, se não
me engano. O gpt, atualmente, utiliza o antlr v2 gerando código C++, e
provavelmente, continuará utilizando esta versão para o branch 1.x.


Agora, estava dando uma olhada rápida na nova versão do antlr (v3) e
notei que, por enquanto, eles suportam/citam os seguintes alvos: C,
C#, Java, Objective-C, Python, Ruby.

Ruby parece não estar completo. Python acho que está em beta.
Objective-C eu não sei, e também não sei se é compatível com o
gcc-objc, ou se é muito ligado à implementação do Mac. C#, acho que
está maduro. C também. Java, acredito que seja o alvo principal, e
referência para as outras.

Com isso em vista, e sabendo que pretendo utilizar o antlr3 para o gpt
v2, temos que decidir qual destas linguagens vamos utilizar para o
novo compilador do gpt.

Estava pensando em escolher, inicialmente, uma dessas linguagens
apenas para bootstrap. Depois, poderiamos tentar criar um target do
ANTLR para o g-portugol. Daí, escreveriamos o compilador com ANTLR
gerando código g-portugol, e teriamos um "self-host compiler". Acho
que seria um bom exercício para a linguagem.
