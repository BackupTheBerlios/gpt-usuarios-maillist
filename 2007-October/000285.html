Eu coloquei como opcional tendo em vista as unidades escritas para
serem bibliotecas. Inicialmente, achei estranho obrigar a declaração
de algoritmo em tais arquivos.

> Entendi a questão do flexível.
>
> A minha dúvida seria a seguinte:
>
> // Trecho em portugol...
> variáveis
>     x : inteiro;
>     y : inteiro;
> fim-variáveis
> início
>     x := 10;
>     y := 20;
>     x + y; // o que faria esse trecho de código ????
> fim
>

A principio, faria a soma. Em um segundo momento, o respectivo código
seria descartado, por não ter efeito colateral. É o mesmo que na
linguagem C, imagino.

Por outro lado, para ilustrar um caso prático, isso seria interessante
se fizessemos um REPL (read eval print loop), como os populares shells
do ruby ou python. Permitir expressões como enunciados é interessante
para mostrar o resultado das expresssões na medida em que o usuário as
escreve. Mas para termos este benefício, teriamos que projetar a VM e
a linguagem para se adequarem bem a modificação/adição de códigos em
tempo de execução, algo que não sei se é viável pela natureza rígida
da linguagem. Para fazer essas adequações, acho que sairíamos do
"portugol canônico".

> Poderia ser um "fim" apenas, mas acho estranho porque sempre tem a seqüência
> "estrutura...fim-estrutura".

hehehe. Inspirado no seu bicão:

caso x seja
    1:
        <comandos>
    fim-opção
    2:
        <comandos>
    fim-opção
    senão
        <comandos>
    fim-opção
fim-caso

E mais um:

caso x seja
    1:
        <comandos>
    sair;
    2:
        <comandos>
    sair;
    senão
        <comandos>
    sair;
fim-caso

Poderiamos adicionar a palavra-chave "sair" como análoga ao "break". E
serviria para laços também. Mas, claro, o problema da (in)coerência em
relação às outras estruturas ainda persiste.
