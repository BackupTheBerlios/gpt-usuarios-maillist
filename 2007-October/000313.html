Voltando a pendência do acesso ao kernel, gostaria de solucionar isso.

Utilizando a função leia(literal) como exemplo, vejo as seguintes
opções para implementá-la:

-Nota: para evitar maiores confusões, suponha, para este email, que
início-asm seja inline assembly, e início-gasm seja inline
gptassembly.


#1 - inline assembly

função leia(ref lit : literal)
 tam : inteiro
início
  início-asm
    mov eax, 3
    mov ebx, 0
    mov ecx, [lit]
    mov edx, [tam]
    int 80h

    mov ebx, [lit]
    mov [ebx+eax-1], byte 0
  fim-asm
fim

Implicações que observo:

a) a VM teria que ser um pouco esperta na emissão de código para lidar
com as referências à variáveis como "lit", especialmente considerando
que um literal provavelmente não seja mapeável diretamente para um
char*.

b) Acho que o código e a VM ficam muito sensíveis. Tem que se conhecer
bem assembly, gptassembly e o casamento das duas para funcionar.

c) Acho que colocar assembly direto pode impedir otimizações a nível
de opcode, quando blocos de código asm estiverem envolvidos...

Enfim, não estou muito certo dessa abordagem, embora seja a minha
preferida no diz respeito a mostrar para o usuário a implementação de
uma função "até o talo".


#2 - Sugestão anterior do Alex

Num email anterior, vc tinha escrito isso:

push "cria" // função
push "thread" // "classe"
libcall

Poderia elaborar mais?


#3 - Biblioteca de runtime

Acho que é minha interpretação/entendimento da opção #2.

Suponha a existência de uma biblioteca, libgptruntime.so, escrita em
C, com as seguintes funções:

readline
getuid
mkdir
connect
...etc

(se em inglês, ou português, e que nomes serão, decidiriamos depois)

A implementação (em C) de readline, por exemplo, poderia ser assim:

gpt_literal readline(gpt_inteiro fd) {
  //.....
}

Então, poderiamos escrever leia(literal), em g-portugol, da seguinte forma:

função leia( ref lit : literal)
início
  início-gasm
     push_int STDIN //constante declarada em algum lugar...
     libcall libgptruntime.so, readline
     pop lit
  fim-gasm
fim
