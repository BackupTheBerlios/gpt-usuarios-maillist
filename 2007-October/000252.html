Ok, eu experimentei um pouco aqui duas abordagens para resolver este problema.


* Abordagem #1 - não dar suporte em alto nível para manipulação de
argumentos variáveis

Aqui, poderiamos aceitar a sintaxe:

função imprima( ...)

Mas não dariamos suporte, em alto nível, para manipular os argumentos
passados. Isso significa que o corpo de "imprima" teria que ser
escrito com gptassembly inline, para acessar a pilha de argumentos
diretamente e checar os tipos de cada um manualmente.


* Abordagem #2 - Oferecer meios especiais para manipular os argumentos variáveis

Aqui, supús uma entidade "ARGV", que seria um array contendo todos os
argumentos. Então, ARGV[0] seria o número total de argumentos
passados. Então, escrevi isso:

função imprima( ... )
 numargs : inteiro;
 i       : inteiro;
início
  numargs := ARGV[0];
  para i de 1 até numargs-1 faça
    imprima(ARGV[i]);
  fim-para
fim

função imprima(l:literal) //...
função imprima(r:real)    //...
função imprima(i:inteiro) //...


Mas não me parece tão boa idéia, pois a complexidade para checar tipos
e escolher a versão correta de "imprima" seria passada para o
compilador, ou para os bytecodes.

Então, procurando fazer a ligação estática de nomes, pensei em fazer
com que ARGV fosse um array com elementos especiais. Cada elemento do
array seria uma instância de uma estrutura com dois atributos "tipo" e
"valor":

estrutura
  tipo : literal;
  valor : <*>;
fim-estrutura

O tipo do valor, aqui, não pode ser definido estaticamente, daí ser um
caso especial. Então, re-escrevi "imprima" da seguinte forma:

função imprima( ... )
 numargs : inteiro;
 i       : inteiro;

 lit : literal;
 r   : real;
início
  numargs := ARGV[0].valor;

  para i de 1 até numargs-1 faça
    caso ARGV[i].tipo seja
      "literal":
         lit := ARGV[i].valor;
         imprima(lit);
         sair;
      "real":
         r := := ARGV[i].valor;
         imprima(r);
         sair;
      //etc
    fim-caso
  fim-para
fim

Dessa forma, todas as chamadas a "imprima" são resolvidas em tempo de
compilação. Mas em termos de avaliação de tipos, isso é tão inseguro
quanto em C. E eu ainda não sei qual seria a repercussão de uma
atribuição incorreta envolvendo tipos incompatíveis, e nem como isso
tudo se resolve durante a compilação e geração de bytecodes.
