Pensando rapidamento sobre isso, eu acho teria implicações sobre a VM sim,
mas não sei dizer o quanto agora sem uma análise maior.

Saliento que a área de dados é uma estrutura simples, apenas uma sequência
de bytes, similar a área de dados de um processador real. Por exemplo, se
tivermos 1 variável caractere (c1) e duas variáveis inteiras (i1 e i2),
teríamos  a seguinte representação na área de dados:

0000: 00 // Endereço 0000, um byte, alocado para c1
0001: 00 00 00 00 // Endereço 0001, 4 bytes, alocados para i1
0005: 00 00 00 00 // Endereço 0005, 4 bytes, alocados para i2

Por isso essa questão de tamanhos dinâmicos pode impactar bastante no que já
está definido para a versão 2, tanto em termos de armazenamento como
opcodes.

Pensando um pouquinho mais, haveria algum problema se a gente sempre
utilizasse o maior tipo disponível na arquitetura para armazenar uma
determinada informação ? Por exemplo, real utilizaria double, int usaria
int64, ...

Eu só não sei dizer se isso resolveria completamente o problema que você
relatou porque a minha solução só resolve para os casos onde seja possível
mapear a informação para um tipo nativo da máquina.

O 983758604983758604983758604983758604983758604983758604 que você colocou
acima não dá pra mapear em um int64...
