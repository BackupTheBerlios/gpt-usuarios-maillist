Complementando um pouco mais...

Através do exemplo abaixo eu tentei criar o GptAssembly resultante
utilizando as 2 abordagens citadas.


estrutura est
    x: inteiro;
    y: inteiro;
fim-estrutura
variáveis
    i: inteiro;
    l: literal;
    e: est;
    m1: matriz[] de inteiros;
    m2: matriz[5] de inteiros;
fim-variáveis
início
    i := 10;
    l := "teste";
    e.x := 1;
    e.y := 2;
    m2[0] := 10;
    m2[2] := 20;
    i := m2[2];
    m1 := m2;
fim

A abordagem 1 gerou uma sequência confusa e cheia de detalhes (isso sem
contar o que eu devo ter esquecido de colocar):

// inicialização dos dados
balloc l, 0 // aloca espaço para o literal
balloc m1, 0 // aloca espaço para m1
balloc m2, 20 // 5 * sizeof(int)
balloc e, 8   // sizeof(est)

iset i, 10           // i := 10;
bcopy l, "teste", 5  // l := "teste";
bset e:0, 1, 4       // e.x := 1;
bset e:4, 2, 4       // e.y := 2;
bset m2:0, 10, 4     // m2[0] := 10;
bset m2:8, 20, 4     // m2[2] := 20;
bget i, m2:8, 4      // i := m2[2];
bcopy m1, m2, 20     // m1 := m2;

// destruição dos dados
bfree l
bfree m1
bfree m2
bfree e
Observações: bcopy verifica se o destino tem espaço para copiar os dados. Se
não tiver ele aloca espaço antes de copiar. Se já estiver alocado algo, ele
desaloca antes de alocar novamente.

A abordagem 2 gerou uma sequência mais "legível"...

m1alloc m1, 4, 0
m1alloc m2, 4, 5

iset i, 10       // i := 10;
sset l, "teste"  // l := "teste";
isetv e:0, 1     // e.x := 1;
isetv e:4, 2     // e.y := 2;
m1set m2:0, 10   // m2[0] := 10;
m1set m2:2, 20   // m2[2] := 20;
m1get i, m2:2    // i := m2[2];
m1copy m1, m2    // m1 := m2;

m1free m1
m1free m2



Observações: malloc (matrix alloc) define que, para uma variável x, ela irá
suportar um tipo de tamanho Y, com Z elementos iniciais. Teoricamente, com
isso, podemos ter matriz composta de estruturas definidas pelo usuário.

As minhas impressões sobre isso...

Com a abordagem 1 (a que gerava mais opcodes) eu achei a geração "chata" e
"cheia de detalhes". O tipo da coisa que para alguém programar em
GptAssembly depois vai ter que ter uma "lista de regras".

Já com a abordagem 2, o código gerado está mais legível para nós, meros
mortais :-)

Além do que, com a abordagem 2, podemos estender (teoricamente facilmente) o
suporte para termos matrizes de tamanhos variáveis...

Claro, m1 significa "uma dimensão". Teríamos que ter o m2, m3, .... mas vou
tentar ver alguma forma de simplificar isso... Se a gente não conseguir
simplificar a gente limita o número máximo de dimensões (2 o 3).

Bom, claramente o meu voto vai para a abordagem 2 :-)
